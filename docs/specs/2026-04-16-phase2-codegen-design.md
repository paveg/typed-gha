# Phase 2: `gha add` — action.yml codegen

## Overview

Extend `@typed-gha/cli` with a `gha add <action-ref>` subcommand that fetches an
action's `action.yml`, parses its inputs/outputs schema, infers TypeScript types,
and emits a typed wrapper file using the same `makeAction` factory shape as the
Phase 1 handwritten wrappers. Users commit the generated file and import it with
a relative path — no separate package install or tsconfig changes required.

## CLI interface

```
gha add <action-ref> [--dir <path>]
```

### `<action-ref>` formats

| Format | Example | Resolution |
|---|---|---|
| Remote (owner/name@ref) | `actions/checkout@v4` | GitHub API or raw.githubusercontent.com |
| Local file | `./actions/my-action/action.yml` | `fs.readFile` |
| Local directory | `./actions/my-action/` | Look for `action.yml`, then `action.yaml` |

Ref is **required** for remote actions. `gha add actions/checkout` (no `@ref`) is an
error: `"ref is required (e.g., actions/checkout@v4)"`.

### `--dir`

Output directory for the generated wrapper. Default: `.github/typed-gha-actions/`.
Created automatically if it does not exist.

### Output file naming

`{owner}-{name}.ts` — e.g., `actions/checkout@v4` → `actions-checkout.ts`.

For local actions: derive from the path. `./actions/my-action/action.yml` →
`actions-my-action.ts` (parent dir + action dir).

### Overwrite semantics

Existing files are overwritten unconditionally (idempotent). The generated header
`// Generated by typed-gha from {source}. Do not edit by hand.` signals that the
file should not be manually edited.

## Resolver

Unified interface for fetching action.yml from any source:

```ts
type ActionSource = {
  raw: string            // action.yml content
  ref: string            // 'actions/checkout@v4' or './path/to/action'
  repoUrl: string | null // GitHub URL for @see links (null for local)
}
```

### Local resolution

- Input starts with `./`, `../`, or `/`.
- If path ends in `.yml` or `.yaml`, read directly.
- If path is a directory, try `action.yml` then `action.yaml`. Error if neither exists.
- `repoUrl` is `null`.

### Remote resolution

- Input matches `{owner}/{name}@{ref}` pattern.
- Primary: `gh api repos/{owner}/{name}/contents/action.yml?ref={ref}` → base64 decode.
  - Falls back to `action.yaml` if `action.yml` returns 404.
- Fallback (if `gh` CLI not available): `fetch('https://raw.githubusercontent.com/{owner}/{name}/{ref}/action.yml')`.
- `repoUrl`: `https://github.com/{owner}/{name}`.

### Error cases

- No `action.yml` or `action.yaml` found → `Error: No action.yml found at {ref}`
- Missing ref on remote → `Error: ref is required (e.g., actions/checkout@v4)`
- Network failure → propagated as-is (CLI catch-all displays it)

## Parser

Extracts structured metadata from the raw YAML content.

### Parsed structure

```ts
type ParsedAction = {
  name: string
  description: string
  inputs: ParsedInput[]
  outputs: ParsedOutput[]
  runsUsing: string // 'node20' | 'composite' | 'docker' etc.
}

type ParsedInput = {
  key: string          // original YAML key (e.g., 'fetch-depth')
  description: string
  required: boolean
  default?: string | boolean | number
  inferredType: 'string' | 'boolean | string' | 'number | string'
}

type ParsedOutput = {
  key: string
  description: string
}
```

### Type inference heuristics

Applied to each input based on its `default` value:

| Signal | Inferred TS type | Example |
|---|---|---|
| `default: true` or `default: false` | `boolean \| string` | `lfs: { default: false }` |
| `default` is a number (0, 1, etc.) | `number \| string` | `fetch-depth: { default: 1 }` |
| Everything else | `string` | Most inputs |

Enum inference from `description` text is explicitly deferred — too fragile for
automated generation. Users who want narrower types should write a manual wrapper
(same as Phase 1).

### Handling all runs.using variants

All three action types (node, composite, docker) are parsed identically: read
`inputs` and `outputs` at the top level. `runs.using` is captured for metadata
only. Steps inside composite actions are ignored.

## Code generator

Transforms `ParsedAction` into a TypeScript source string.

### Output shape

```ts
// Generated by typed-gha from actions/checkout@v4. Do not edit by hand.
import { makeAction } from '@typed-gha/core/actions/_factory'

/**
 * Check out a repository
 *
 * @see https://github.com/actions/checkout
 */

export type CheckoutInputs = {
  repository?: string
  ref?: string
  token?: string
  /** @defaultValue true */
  'persist-credentials'?: boolean | string
  /** @defaultValue 1 */
  'fetch-depth'?: number | string
  // ... remaining inputs
}

export type CheckoutOutputs = {
  ref: string
  commit: string
}

export const checkout = makeAction<
  'actions/checkout@v4',
  CheckoutInputs,
  CheckoutOutputs
>('actions/checkout@v4')
```

### Naming conventions

- **Type names**: PascalCase of the action name + `Inputs`/`Outputs`.
  `setup-node` → `SetupNodeInputs`, `SetupNodeOutputs`.
- **Function name**: camelCase of the action name.
  `setup-node` → `setupNode`, `configure-aws-credentials` → `configureAwsCredentials`.
- **Input/output keys**: preserved as-is from action.yml (kebab-case stays kebab-case,
  quoted in the TS type).

### TSDoc on generated code

- Top-level function JSDoc: action's `name` + `description` from action.yml.
  `@see` link to `repoUrl` if available.
- Per-input field: `description` from action.yml as one-line JSDoc.
  `@defaultValue` tag if a default exists.
- Per-output field: `description` from action.yml.

### makeAction import path

Requires a new subpath export in `@typed-gha/core/package.json`:

```json
"exports": {
  ".": { ... },
  "./actions": { ... },
  "./actions/_factory": {
    "types": "./dist/actions/_factory.d.ts",
    "import": "./dist/actions/_factory.js"
  }
}
```

This is the only change to `@typed-gha/core` in Phase 2.

### All inputs are optional

Even `required: true` inputs are typed with `?` in the generated wrapper.
Rationale: the `with` object itself is optional on `UsesStep`, so TypeScript
cannot enforce required-ness at the type level without breaking the shared
factory signature. Same decision as Phase 1.

## Module structure (new/changed files)

### New files in `packages/cli/src/`

| File | Responsibility |
|---|---|
| `add.ts` | `runAdd(opts)` orchestrator: resolve → parse → generate → write |
| `resolve.ts` | Action.yml fetching (local + remote) |
| `parse.ts` | YAML parse + inputs/outputs extraction + type inference |
| `generate.ts` | ParsedAction → TypeScript source string |

### Changed files

| File | Change |
|---|---|
| `packages/cli/src/args.ts` | Add `add` subcommand + `--dir` flag parsing |
| `packages/cli/src/index.ts` | Route `add` command to `runAdd` |
| `packages/core/package.json` | Add `./actions/_factory` subpath export |

## Test strategy

All tests are vitest, following TDD (red-green-refactor).

### `test/parse.test.ts`

- Boolean default → `boolean | string` inference
- Number default → `number | string` inference
- String/no default → `string`
- Required field parsed correctly
- Outputs extracted
- Empty inputs/outputs handled
- All three `runs.using` variants (node, composite, docker) parse identically

### `test/generate.test.ts`

- Snapshot of generated .ts for a known ParsedAction
- camelCase function naming (`setup-node` → `setupNode`)
- PascalCase type naming
- Boolean/number inferred types appear in output
- Header comment includes source ref
- `@defaultValue` tags present
- `@see` link present when repoUrl provided, absent when null

### `test/resolve.test.ts`

- Local: tmpdir with action.yml → reads content
- Local: dir input → finds action.yml
- Local: dir with only action.yaml → falls back
- Local: dir with neither → throws
- Remote: not tested with real network in unit tests (integration test below)

### `test/add.test.ts` (integration)

- End-to-end: tmpdir + local action.yml fixture → `runAdd` → file created with correct content
- Overwrite idempotency: run twice, assert identical output
- `--dir` flag: output lands in specified directory
- Missing ref error

### Integration test with real actions (optional, CI-gated)

- `gha add actions/checkout@v4` against the real GitHub API
- Compare generated output against the Phase 1 handwritten `checkout.ts` for structural compatibility
- Gated behind `TYPED_GHA_INTEGRATION=1` env var (not run in normal `pnpm test`)

## Non-goals for Phase 2

- Enum inference from description text
- `gha update` / `gha remove` subcommands
- tsconfig `paths` auto-configuration
- Barrel `index.ts` auto-generation
- package.json config (`typed-gha.actionsDir`)
- Private action auth configuration (uses ambient `gh` auth)
